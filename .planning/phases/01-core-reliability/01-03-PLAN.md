<objective>
Refactor `aws_copier/core/folder_watcher.py` to fix three tightly-coupled defects: (1) ASYNC-01 — replace the broken `self.event_loop.call_soon_threadsafe(asyncio.create_task, coro)` bridge with `asyncio.run_coroutine_threadsafe(coro, self.event_loop)`, which actually schedules real-time watchdog events onto the asyncio loop on Python 3.10+; (2) ASYNC-05 — update the `FileChangeHandler.__init__` type annotation from the private `asyncio.BaseEventLoop` to the public `asyncio.AbstractEventLoop`; (3) IGNORE-03 — delete the diverging local `ignore_patterns` set and `_should_ignore_file` method, and consume the `IGNORE_RULES` singleton from Plan 01 instead.

Purpose: Today, saving a file in a watched folder does NOT trigger an upload until the next 5-minute scan cycle, because `call_soon_threadsafe(asyncio.create_task, coro)` on Python 3.10+ invokes `create_task` with no running loop reference inside the main thread and silently drops the coroutine. `run_coroutine_threadsafe` is the correct API: it returns a `concurrent.futures.Future` and handles the thread-to-loop handoff. The `_should_ignore_file` / `ignore_patterns` duplication is the reason dot-files currently bypass the 5-minute scan filter — unifying through `IGNORE_RULES` closes that asymmetry.

Output: Modified `aws_copier/core/folder_watcher.py` (6 edits), modified `tests/unit/test_folder_watcher.py` (assertions swapped from `call_soon_threadsafe` to `run_coroutine_threadsafe`, plus a new behaviour-proving ASYNC-01 test).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Watchdog OS thread → asyncio event loop | The FileSystemEventHandler runs inside watchdog's Observer thread. The asyncio loop runs in the main thread. Any correctness bug at this boundary silently loses real-time events — the primary reliability bug this plan closes. |
| Filename → ignore-rule evaluator | File paths from watchdog events are checked by IGNORE_RULES.should_ignore_file before being scheduled. If the bridge fires before the ignore check, a sensitive file could be scheduled for upload even while IGNORE_RULES would have blocked it. Current code (and post-fix code) performs the ignore check FIRST, then schedules — the correct ordering. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering (data correctness) | `call_soon_threadsafe(asyncio.create_task, coro)` bridge silently drops events on Python 3.10+ | mitigate | Replace with `asyncio.run_coroutine_threadsafe(coro, self.event_loop)`. Tested by `test_on_any_event_uses_run_coroutine_threadsafe`. |
| T-03-02 | Information Disclosure | `.env` and SSH keys bypass ignore filter during the 5-minute scan (because `FileListener._should_ignore_file` does not block dot-files, unlike the handler) | mitigate | Delete local `_should_ignore_file` and `ignore_patterns`; consume `IGNORE_RULES` singleton. After Plan 04 also applies IGNORE_RULES on the scan path, the asymmetry is fully closed. This plan closes the handler side. Tested by `test_on_any_event_skips_ignored_file_via_ignore_rules`. |
| T-03-03 | Denial of Service | Calling `.result()` on the returned Future from `run_coroutine_threadsafe` inside the watchdog thread would deadlock the watchdog → asyncio handoff | accept | Comment in code warns against calling `.result()`. Pattern is fire-and-forget (the Future is discarded). Documented in Pitfall 2 of RESEARCH.md. |
| T-03-04 | Repudiation | Events lost silently leave no audit trail | accept-with-monitoring | `_stats["events_processed"]` in `FolderWatcher` (already exists) records the count; Plan 04 will add `ignored_files` tracking. Event loss is detectable via stats deltas at the application level. |
| T-03-05 | Elevation of Privilege | A symlink inside a watched folder pointing to `~/.ssh/` could expose keys via the scan path | mitigate | `IGNORE_RULES.should_ignore_dir` (Plan 01) returns True for symlinks. Plan 04 will wire it into the scan recursion. This plan's ignore hook only covers file events, but `IGNORE_RULES.should_ignore_file` blocks the common sensitive filenames (`id_rsa`, `*.pem`, `*.key`) directly. |
</threat_model>

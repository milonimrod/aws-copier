# Quick Task 20260426-b540: Fix MD5 Hashing Bottleneck

## What Changed

**`aws_copier/core/file_listener.py`**

- Added `_calculate_md5_sync` — a regular (non-async) method that opens the file with stdlib `open()`, reads in **1MB chunks** (`iter(lambda: f.read(1048576), b"")`), and returns the hex digest
- Replaced `_calculate_md5` body: was an `aiofiles` chunk loop (8192 bytes per chunk, `hasher.update()` on the event loop thread); now a single `await asyncio.to_thread(self._calculate_md5_sync, file_path)` call
- `aiofiles` import kept — still used in `_load_backup_info` and `_update_backup_info`

## Why This Was Slow

The old loop used `aiofiles` which offloads disk reads to a thread pool but delivers each 8KB chunk back to the event loop where `hasher.update(chunk)` ran synchronously. With ~50 concurrent coroutines each doing 512+ round-trips per file, the event loop was continuously context-switching for tiny chunks of CPU work — achieving ~1.3 files/second.

## Why The Fix Works

`asyncio.to_thread` moves the entire read+hash loop into a thread pool worker:
- No per-chunk event loop round-trips (128x fewer than 8KB → 1MB)
- CPython's `hashlib` releases the GIL during C-level hashing, so multiple threads genuinely parallelize
- Event loop is fully free while hashing runs — can handle uploads, folder walks, etc. concurrently

The existing `md5_semaphore(50)` still caps thread pool concurrency to prevent thread explosion.
